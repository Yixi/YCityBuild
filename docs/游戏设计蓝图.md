## 背景与目标

现有仓库是一个基于 **Babylon.js 9 + TypeScript（非严格）+ webpack 5 + pnpm** 的 3D 城市建造 **Demo**：渲染 100×100 地图、随机生成树、按预设数据绘制道路，并支持**点击建路**（道路外观按四邻居动态计算）。

它提供了一套很好的骨架——**网格地图 + 道路拼接 + 点击建造 + 邻居动态计算**——但「经营游戏的灵魂」尚未具备：

- 没有时间驱动（`runRenderLoop` 无 tick 钩子）；
- 没有金钱 / 人口 / 资源 / 需求；
- 没有公用事业网络；
- 没有 UI / HUD（仅一个显示 FPS 的 DOM 元素）。

本文档是将其做成一款**完整城市模拟经营游戏**的总纲。项目方向定为：

- **游戏定位**：硬核深度模拟（Cities: Skylines 路线）——精细交通流、独立水电管网、区域政策、公交线路。
- **建造范式**：分区自动生长（zoning）——玩家修路 + 刷 R/C/I 分区，系统按需求自动「长」建筑。
- **美术风格**：全程沿用程序化几何（Box / Merge / 实例化），零美术资源即可跑通所有系统。

> 本蓝图体量大、是长期路线。落地策略是**先重构地基、再做最小可玩闭环（MVP）、然后逐阶段叠加硬核系统**，每阶段都可独立运行与验证。

## 一、核心体验与玩法循环

**玩家幻想**：你是市长。在一张空地上规划路网、划定功能区、铺设管网与服务，平衡**收支、需求、交通、环境、民意**，让城市从村镇成长为大都会。

**三大体验支柱**：

1. **规划的因果感**——每条路、每个分区都会通过交通 / 地价 / 污染 / 服务**涟漪式**影响周边。
2. **系统的相互纠缠**——交通堵 → 商业衰 → 税收降 → 服务砍 → 民意跌，硬核乐趣来自牵一发动全身。
3. **从无到有的成长**——人口里程碑解锁新建筑与政策，城市天际线随密度自然「长高」。

**核心玩法循环（The Loop）**：

```
规划路网 ──▶ 划定 R/C/I 分区 ──▶ 铺电/水/服务 ──▶ 建筑自动生长、人口入住
   ▲                                                          │
   │                                                          ▼
扩张/改造 ◀── 用税收再投资 ◀── 收税、平衡收支 ◀── 应对交通/污染/民意问题
```

每一圈循环的「卡点」（堵车、缺电、污染、赤字、民怨）正是硬核模拟的玩点，玩家通过**升级路网 / 增容管网 / 调整政策 / 优化布局**解题。

## 二、玩法系统设计（游戏设计层）

下列 14 个系统构成完整游戏。标注 **[MVP]** 的是第一个可玩切片必含的最小版本，其余为后续阶段叠加。每个系统给出「机制 / 与其他系统的耦合 / 数值要点」。

### 1. 时间与节奏系统 [MVP]
- **机制**：游戏内时钟以「tick」为最小步长推进，聚合为 **日 → 月 → 年**。跨月触发经济结算、跨日触发普查 / 交通重算等周期事件。支持**暂停 / 1x / 2x / 3x** 变速。
- **耦合**：所有模拟系统的心跳源。
- **数值要点**：建议 `ticksPerDay≈30`、`SIM_DT=1/30s`（详见技术架构 §三.2）。重经济按月算、轻状态按 tick 算，避免每帧重算。

### 2. 经济系统 [MVP]
- **机制**：单一货币。**支出** = 建造一次性费用 + 每月维护费（道路 / 管网 / 服务建筑按规模计）；**收入** = 分区税收（R/C/I 各一档税率）。月度结算资产负债。后续加**贷款 / 债券、预算分项拨款**（按系统调拨经费比例）。
- **耦合**：维护费随城市规模线性增长，税收随人口 / 商业繁荣增长——收支平衡是核心难度旋钮。
- **数值要点**：初始启动金（如 ¥70,000）；税率默认 9%~12%，可调；破产 = 负债超阈值触发警告 / 限制，而非直接 game over。

### 3. 人口与 RCI 需求系统 [MVP]
- **机制**：维护三类需求标量 **R（住宅）/ C（商业）/ I（工业）**，是分区生长的驱动力。需求由就业-居住平衡、税率、可建空间、交通便利、地价等推导（经典 SimCity「RCI 条」）。人口有**就业率、平均收入、教育水平、幸福度**等聚合指标。
- **耦合**：I 提供岗位 + 货物 → C 需要岗位 + 货物 + 顾客 → R 提供劳动力 + 消费力，三者动态再平衡。
- **数值要点**：需求为有上下限的反馈量；空置率过高则需求转负（建筑废弃）。

### 4. 分区与建筑自动生长系统 [MVP]
- **机制**：玩家只**刷分区**（R/C/I + 后续密度低 / 中 / 高），不直接盖楼。模拟内核按条件让格子上的建筑**生长 → 升级 → 衰败 → 废弃**（状态机）。生长**必要条件**：临路（复用道路邻居读取）+ 通电 [MVP] + 对应需求为正；**升级条件**：地价 / 服务 / 需求达标；**衰败**：断电断路 / 污染过高 / 需求转负。
- **耦合**：消费 RCI 需求、占用电力 / 水、贡献税收与人口、受地价 / 污染 / 服务 / 交通影响。
- **数值要点**：每 tick 限量生长（生长预算 K 格）+ 概率 roll（确定性 PRNG），模仿城市渐进铺开；迟滞阈值防止反复横跳。

### 5. 道路与交通系统 [MVP 含路；交通流见阶段 2]
- **机制**：道路是路网骨架，建筑必须临路才能生长。**[MVP]** 复用现有道路拼接与动态外观。**[阶段 2]** 升级为**硬核交通流**：路网抽象为图，按通勤 OD（居住 ↔ 就业 ↔ 商业）用**重力模型 + 静态容量分配**估算各路段流量与拥堵，**每游戏日重算一次**缓存。多道路等级（小路 / 大道 / 单行 / 高速）。
- **耦合**：拥堵 ↓ 商业销售、↑ 污染、↓ 地价、↓ 服务到达率 → 反哺生长与民意。
- **数值要点**：用「路段容量 vs 分配流量」的比值表征拥堵，**不做逐车 A***（性能见技术架构 §三.5）。

### 6. 公用事业：电力网络 [MVP]
- **机制**：发电厂（或后续光伏 / 风电）产生电力，沿**相邻的路 / 电网**传导（连通性用 union-find）。每个连通分量是一本**供需账本**：`总供给 ≥ 总需求` 则全供，不足则按确定性优先级**轮流限电**。无电的建筑**停止生长 / 停业 / 逐步废弃**。
- **耦合**：生长前置条件；发电厂产污染、占地、有维护费。
- **数值要点**：每类建筑有耗电量；发电厂有装机容量与单位维护成本；预留「电网 vs 沿路供电」二选一（MVP 先沿路供电，省一套管线交互）。

### 7. 公用事业：水 / 污水网络 [阶段 3]
- **机制**：水厂抽水 + 水管网络供水；污水需排放 / 处理，否则污染水体与地价。与电力同构（连通 + 供需账本），但需独立铺设**水管**（地下层）。
- **耦合**：缺水同样抑制生长；污水处理不足 → 污染飙升。

### 8. 公共服务与覆盖系统 [阶段 4]
- **机制**：消防、警察、医疗、教育、垃圾、公园 / 绿化等服务建筑，各有**覆盖半径 / 服务容量**（沿路距离衰减）。覆盖范围内的建筑获得对应**服务度**加成。
- **耦合**：服务度 ↑ 民意、↑ 地价、↑ 建筑升级上限；服务不足 → 火灾 / 犯罪 / 疾病事件、地价下跌。
- **数值要点**：覆盖用「沿路距离衰减场」而非欧氏圆，体现路网依赖。

### 9. 环境系统：污染 / 噪音 / 地价 [阶段 3]
- **机制**：工业、交通、发电厂等**源**向周边格子**扩散**污染 / 噪音（网格上做卷积 / BFS 扩散场）。**地价**由服务、绿化、景观、（负向）污染 / 噪音 / 拥堵综合而成。
- **耦合**：污染 ↓ 住宅吸引力、↓ 地价；地价决定建筑可达密度与税收。
- **数值要点**：污染 / 地价是逐 tick 缓慢演化的标量场，避免突变。

### 10. 政策与法规系统 [阶段 6]
- **机制**：全局或分区域的政策开关（如禁工业污染、节能补贴、限速、垃圾分类、差异化税率）。每条政策有收益与成本 / 副作用。
- **耦合**：作为玩家调节各系统的「旋钮」，提供策略深度。

### 11. 公共交通系统 [阶段 6]
- **机制**：玩家铺设公交 / 地铁**站点 + 线路**，分担道路交通、扩大服务可达性。线路有运力、票价收入与运营成本。
- **耦合**：缓解拥堵 → 提升商业 / 地价；运营是新的收支项。

### 12. 灾害与随机事件系统 [阶段 7]
- **机制**：火灾蔓延、停电连锁、洪水、交通事故、经济波动、市民诉求事件等。考验城市韧性与应急服务。
- **耦合**：检验服务 / 管网的冗余度，制造动态叙事。

### 13. 进度与解锁系统 [阶段 6]
- **机制**：以**人口里程碑**为主线（村 → 镇 → 城 → 大都会），逐级解锁新建筑、新服务、新政策、地图扩张。辅以成就 / 挑战目标。
- **耦合**：提供长期目标与节奏控制，避免一次性铺满所有系统。

### 14. 数据可视化图层系统（信息视图）[MVP 含 1 个图层]
- **机制**：可切换的热力叠加图层——电力覆盖 [MVP]、交通拥堵、污染、地价、服务覆盖、水覆盖等，在地图上以颜色直观呈现，是硬核模拟的「诊断仪表盘」。
- **耦合**：读各系统的标量场渲染（技术实现见 §三.9）。

## 三、技术架构（工程落地层）

总原则：**模拟逻辑（纯数据、确定性）与渲染表现（Babylon mesh）彻底分离**；**事件 / 脏标记驱动**而非每帧全量扫描；**实例化渲染**根治 draw call 爆炸。

### 1. 分层总览（目标 `src/` 结构）

```
src/
  index.ts          # 入口：Engine/Scene + 启动 SimClock + 渲染循环
  scene.ts          # 组装：map→camera→light→core→renderers→overlay→interaction→ui
  core/             # 模拟内核：SimKernel / GridModel / SimClock / Scheduler / CommandBus / EventBus / selectors
  sim/              # 各模拟系统（纯逻辑，禁止 import BABYLON）：economy/ rci/ growth/ power/ water/ traffic/ service/ environment/ ...
  interaction/      # InputRouter / ToolManager / PickService / PreviewLayer / tools/ / commands/
  render/           # InstanceField / PrototypeRegistry / MaterialLib / 各 Renderer / LodPolicy
  overlay/          # 热力图：OverlayManager / HeatmapTexture / palettes
  selection/        # SelectionService / Highlighter
  camera/           # 增强相机（平移/缩放/旋转分流）
  ui/               # DOM HUD：HudRoot / hud.css / panels/ / widgets/
  persistence/      # 存档序列化
  model/            # 几何原型工厂（沿用并「原型化」road/tree，新增 building/service）
  data/             # store → GridModel 迁移
```

### 2. 确定性模拟 tick 内核
- **固定步长 + accumulator**：在现有 `index.ts:11` 的 `runRenderLoop` 内累加真实 `dt`，按固定 `SIM_DT`（如 1/30s）整数步消费，带 `MAX_STEPS` 上限防「死亡螺旋」。**否决** `onBeforeRenderObservable`（绑帧率、不确定）与 `setInterval`（后台节流、时序错位）。
- **变速**：`speed ∈ {0,1,2,3}` 只缩放「喂入 accumulator 的量」，**绝不改 `SIM_DT`**（改步长会破坏数值稳定与可复现）；2x = 同帧多消费几个固定 step，结果与 1x 跑 N 帧逐位一致。
- **执行顺序（固定的数据依赖拓扑）**：
  `Network(电/水连通) → Service(覆盖场) → LandValue(地价/污染场) → Traffic(路段负载) → RCI(需求) → Growth(生长/衰败) → Economy(月结) → Stats(聚合)`。后系统只读前系统产出；Growth 是唯一改变建筑拓扑者，置于所有「场」计算之后。

### 3. 系统组织：System 数组 + 共享 World + 脏标记（**不用 ECS**）
- **理由**：数据是「规则密集的网格」而非离散异质实体，ECS 的 archetype / 稀疏实体优势用不上；网格天然 SoA，用多张平行 typed-array 即可获得 cache 友好遍历。唯一例外：未来若做逐车 / 逐市民 agent，再为这些移动实体加轻量实体数组。
- **形态**：`store` 单例**保留并演进**为 `World` 载体（现有 `import { store }` 不必全改）。定义统一 `System { update(world, dt) }` 接口 + `Scheduler.tick()`（自增 tick → 推日历 → 按序跑 system → 交换脏集合）。

### 4. 数据模型重构：胖 `IMapData` → 平行 Grid Layers + Building[]
- **现状**：`src/data/store.ts:5` 的 `IMapData = {x,y,mesh?,meshType?,info?}` 把**逻辑与 mesh 混在一格**。
- **重构**：拆成多张薄 typed-array（`tile / zoneType / density / buildingId / roadShape / landValue / pollution / powerNet / supplied / trafficLoad …`）+ 建筑富信息独立结构数组 `Building[]`；mesh 职责收敛到渲染层 `MeshRegistry / InstanceField` 的 `index↔cell` 映射，**逻辑层永不持有 mesh 引用**。
- **为何彻底分离**：① mesh 不可序列化，分离后才能存档；② 确定性要求逻辑只依赖定点 grid，无浮点 mesh 噪声；③ 场扩散在 `Uint8Array` 上做卷积 / BFS 比遍历胖对象快一个量级、GC 友好；④ `sim/` 不 import BABYLON，可在 node 跑纯逻辑单测。
- **迁移策略（低风险）**：先把 `store` 包进 `GridModel` 并**保留旧字段做兼容垫片**（`meshType` 继续可用），渲染逐步从「格子持 mesh」迁到 InstanceField，最后把 `MESH_TYPE`（仅 ROAD/TREE）扩为 `CellKind`（EMPTY / ROAD / ZONE_R/C/I / BUILDING / SERVICE / TREE / WATER…）。**顺带修复** `initMap` 的 `x:floor(value/height), y:value%width` 在非正方形地图错位、字段名 `y` 实为 z 轴的隐患。

### 5. 网络与交通算法
- **电 / 水连通**：**union-find**（路径压缩 + 按秩合并），放一格管线 / 路只做几次近 O(1) 的 union；删除时 union-find 不支持高效删边 → 标记受影响连通分量 dirty，下一 tick 对该分量**局部 flood-fill 重建**（沿用「局部增量」哲学）。**容量**不做真实潮流，每个连通分量是 `supply ≥ demand` 账本，不足按确定性优先级降级——C:S 级合理近似。
- **交通流**（阶段 2，**避免逐车 A***）分三档：L0 只算「建筑到路网可达性」标量（MVP 够用）；**L1 重力模型 + 静态分配**（目标）：路网抽象成图，`flow(i→j) ∝ 产量×吸引/dist²`，一次最短路 + 按容量比例把流量压到路段，**每游戏日跑一次**缓存进 `trafficLoad`，复杂度比逐车 A* 低 2~3 个数量级；L2 仅在可见区域生成少量**装饰车**（不参与逻辑）。

### 6. 渲染性能：Thin Instances 原型分桶 + LOD + 脏区增量
- **根因**：现状每格独立 `MergeMeshes` + `src/model/road.ts:225` 的 `createRoad` 每次 `new StandardMaterial`，城市尺度必然 draw call 与材质双爆炸。
- **方案**：每种原型（道路 5 型、各建筑等级、树…）建**一个原型 mesh + 一个共享材质**，用 **thin instances**（`thinInstanceSetMatrixAt`）摆放——**每原型仅 1 draw call**，且支持单格增删改矩阵，契合「分区零散生长、局部变动」。原型几何置于**原点**注册，旋转 / 平移交给实例矩阵（HORIZONTAL / VERTICAL 共用原型靠 90° 旋转，与现状一致）。**共享材质池 `MaterialLib`** 根除逐格 `new Material`。
- **LOD**：按相机距离 / 缩放分档切换原型（近 = 细节体、中 = 简化 box、远 = 退化为地图色块 / 只留 overlay 贴图）。必要时按 16×16 分块（chunk）做分块剔除与分块脏更新。
- **增量更新**：把 `src/road.ts:103` 的 `rebuildSurroundingArea`（3×3 全量 `dispose` + 重建）**泛化为通用「脏格集协调器 reconciler」**——指令产出 `ChangeSet{dirtyCells, added/removed/upgraded}`，各 Renderer 只消费相关脏项，改一格只改一个实例矩阵，合并一次 `thinInstanceBufferUpdated`，**绝不全量重建**。

### 7. 交互工具状态机
- 用 `InputRouter` 作为**唯一**的 `scene.onPointerObservable` 订阅者，按「按键 + 当前工具」裁决这次手势归**相机**还是归**工具**，再把 down / move / up 派给 `activeTool`。
- 工具策略对象：`RoadTool`（拖拽连续画路，复用 `calculateRoadType`）、`ZoneTool`（矩形框选刷 R/C/I）、`ServiceTool`（放置带覆盖圈预览的设施）、`BulldozeTool`（拆除）、`QueryTool`（点选查询）。
- 写操作走 **Command 模式**（`do/undo`）提交到 `CommandBus`，是模拟内核唯一写入口，产出 `ChangeSet`。`PreviewLayer` 用独立薄实例 / 线框做合法（绿）/ 非法（红）高亮，不污染正式合批 buffer。

### 8. UI / HUD：原生 DOM + CSS 覆盖层（**不引框架、不引 Babylon GUI**）
- **理由**：现项目是纯 TS + 单 `app.html`，`babylonjs` 为 UMD 单文件包（无 `BABYLON.GUI`，引入需 `+数MB`）。DOM + CSS 排版能力强、文字清晰、零额外 GPU 负担、与「非严格 TS、无 JSX」现状最契合。React / Solid 需改 ts-loader 支持 tsx，属过度工程。
- **构建链改动**：起步用**方案 A（零 loader）**——样式写进 `src/ui/hud.css`，`app.html` 用 `<link>` 引入，**完全不改 webpack**；组件多了再升级到 `style-loader + css-loader`。`app.html` 增量：`<div id="hud" style="position:fixed;inset:0;pointer-events:none">`，子面板各自 `pointer-events:auto`（**关键**：空白区把指针透传给 canvas）。
- **HUD 构成**：顶部资源条（钱 / 人口 / 日期 / 收支 / 电·水供需）、底部建造工具栏（修路 / R/C/I / 设施 / 拆除 / 查询）、信息面板（点选详情）、图层切换、变速暂停控件、预算 / 政策弹窗。

### 9. Overlay 热力图：RawTexture（1 像素 = 1 格）
- 用 `BABYLON.RawTexture`（`Uint8Array` RGBA，尺寸 = 地图格数，NEAREST 采样）作为热力数据，贴在**略高于地面（y≈0.02）的独立平面**上（`emissiveTexture` + `disableLighting` + `alpha≈0.6` + `isPickable=false`），**1 draw call**。改一格 = 写 1 个像素，整层 1 次 `update()`；1 万格仅 40KB typed-array + 1 次上传，可每 tick（非每帧）更新，放大到 512×512 仍轻松。**否决**逐格 mesh 着色（draw call 爆）与顶点色（需细分地面 + 重传 buffer）。

### 10. 相机增强与选择
- 保留 `ArcRotateCamera`，**精细化输入分流**：左键彻底让给工具、**右键拖拽旋转**、滚轮缩放（配 `lowerRadiusLimit / upperRadiusLimit` 与 LOD）、WASD / 边缘滚动平移 `camera.target`，倾角限制防翻到地下。拖拽类工具激活手势时 `InputRouter` 临时 `detachControl()` 双保险，根除「建造误触旋转」。
- 选择：`QueryTool` 拾取（建筑为薄实例，命中靠 `pickInfo.thinInstanceIndex` 反查 `index→cell`）→ `SelectionService` 发 `selection-changed` → `InfoPanel` 读 `selectors` 渲染详情；`HighlightLayer` 或发光 marker 高亮。

### 11. 解耦与存档
- **三层数据流**：`Tool → CommandBus（唯一写）→ SimKernel 应用 → ChangeSet → EventBus`。**HUD**（低频聚合量）订阅 `resources-changed / sim-tick` 事件驱动刷新，不在渲染循环里读；**渲染层**订阅 `cells-dirty` 脏标记做增量更新。EventBus 用最简 `Map<event, Set<handler>>` 自实现，无需第三方。
- **存档**：因 mesh 已踢出逻辑层，存档 = 序列化 `World` 纯数据。**只存源真相（拓扑 + 经营态 + PRNG 种子），不存派生态（连通分量 / 场 / 交通负载）**，load 后跑一次 System 重建派生态——存档小、无一致性 bug。确定性（固定步长 + 种子化 PRNG + 固定 System 顺序 + 定点场）额外带来 replay 存档与 golden-test 回归红利。**铁律**：`sim/` 内禁用 `Math.random() / Date.now()`、禁依赖浮点求和顺序、禁依赖 `Map / Set` 非确定遍历序。

## 五、MVP 垂直切片（第一个可玩版本）详述

**一句话目标**：玩家能在空地上 **修路 → 刷 R/C/I 分区 → 建发电厂供电 → 建筑自动生长、人口入住、纳税 → 用税收继续扩张**，跑通城市模拟最小闭环，并已触及两个硬核支柱：**分区自动生长** 与 **电力网络（连通 + 供需）**。

**MVP 必含范围**：
1. **时间**：tick 推进 + 暂停 / 1x / 2x / 3x（系统 §1）。
2. **建造工具**：`RoadTool`（拖拽画路）+ `ZoneTool`（框选刷 R/C/I）+ `BulldozeTool`（拆除）+ `QueryTool`（点选查询）。
3. **分区生长（最小）**：临路 + 通电 + 需求为正 → 生长低密度建筑；断路 / 断电 → 衰败（系统 §4）。
4. **经济（最小）**：建造花钱 + R/C/I 纳税 + 每月结算 + 破产保护（系统 §2）。
5. **人口 / RCI（最小）**：R/C/I 三需求标量驱动生长（系统 §3）。
6. **电力（最小）**：发电厂 + 沿路供电连通（union-find）+ 供需账本 + 缺电抑制生长（系统 §6）。
7. **HUD（DOM）**：顶部资源条（钱 / 人口 / 日期 / 电力供需）+ 底部工具栏 + 变速控件 + 点选信息面板（系统 §14 + 架构 §三.8）。
8. **Overlay**：电力覆盖图层（验证 RawTexture 管线，架构 §三.9）。
9. **存档 / 读档**：序列化 World（架构 §三.11）。

**MVP 明确不含**（留待后续阶段）：交通流模拟、水 / 污水、公共服务、污染 / 地价场、密度升级、政策、公交、灾害、进度解锁。

**MVP 验收闭环**：从空地起步 → 修一条路并刷住宅 / 商业 / 工业区 → 建发电厂 → 观察建筑陆续长出、人口增长、月末税收入账 → 拆掉发电厂或断路验证「断电 → 停业 / 衰败」反馈 → 切换电力图层确认覆盖可视化 → 存档退出、读档恢复。

## 六、验证方式（端到端）

- **逐阶段冒烟**：每阶段 `pnpm dev`（http://localhost:4233）实机操作，对照该阶段「验收」清单逐项确认；`pnpm typecheck` + `pnpm lint` 必须绿（CI 同口径：install → lint → typecheck → build）。
- **性能基线**：右上角 FPS（现成）+ Babylon Inspector 看 **draw call 数**；阶段 0 前后对比验证实例化止血效果；建满一片高密度区压测帧率。
- **确定性回归（硬核模拟关键）**：因模拟确定性，写 **golden test**——固定种子 + 固定操作序列 → 跑 N tick → 断言关键聚合量（人口 / 资金 / 供电）逐位一致；变速 1x / 2x / 3x 结果应一致。`sim/` 纯逻辑可在 node 直接单测（不依赖 BABYLON）。
- **交互 / 视觉**：用 Chrome DevTools 截图核对 HUD 布局与 overlay 着色；手测相机分流（左键建造不误触旋转、右键旋转、滚轮缩放、WASD 平移）。
- **存档**：存档 → 改动 → 读档，断言 World 还原一致（派生态由 System 重算）。

## 七、关键风险与取舍

| 风险 / 决策点 | 取舍 |
|---|---|
| 交通流性能 | **不做逐车 A***，用重力模型 + 静态分配，每日重算缓存；精度换性能 |
| 渲染规模 | thin instances + 共享材质 + LOD + 分块；从阶段 0 就止血，勿等问题爆发 |
| 数据重构爆炸半径 | 直接读 `store.map` 的仅 `road.ts` / `trees.ts` 两处，**重构面小**；用兼容垫片渐进迁移 |
| 确定性被破坏 | `sim/` 铁律：禁 `Math.random / Date.now`、禁浮点顺序敏感、禁 `Map / Set` 非确定遍历；用种子化 PRNG |
| 范围蔓延 | 严守 MVP 边界，硬核系统按阶段叠加，每阶段保持可运行 |
| 非严格 TS | 沿用 `strict:false` 风格（CLAUDE.md 要求），勿擅开 `strictNullChecks` |
| 美术成本 | 全程沿用**程序化几何**（Box / Merge / 实例化），零美术资源即可跑通所有系统；后续如需 `.glb` 模型再加 loader |

## 八、实施起步建议

落地顺序建议：`core/`（SimClock + EventBus + CommandBus 地基）→ `render/InstanceField`（先把路 / 树实例化，止血 draw call 与材质）→ `interaction/`（状态机替换 `clickToBuildRoad`）→ `camera/` 增强 → `ui/` DOM HUD → `overlay/` 热力图 → `selection/`。每步独立验证、不破坏现有运行。

推荐以**阶段 0（架构地基，行为不变）**作为第一个可提交里程碑，再进入 MVP（阶段 1）。阶段 0 内部亦可进一步拆小步推进，最快的可见起点是 **tick 内核（SimClock + 变速 + 游戏内日期显示）**——让「时间跑起来」，再逐步铺开数据层重构与实例化。
