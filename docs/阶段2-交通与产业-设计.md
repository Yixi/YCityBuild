# 阶段 2 设计文档 — 真实模型 · 通勤交通 · 三级产业供应链

> **定位**：在阶段 0/1（已落地、可玩）之上，实现三件事：① 用真实低多边形模型替代方块、让建筑多样化；② 真实通勤交通（居民从住宅出发去工作/购物再回家）；③ 三级产业供应链与货运物流。
> **已确认方向**：通勤采用「统计行程 + 代表性车辆」；供应链做「原料→工厂→商业→居民」三级；模型用 Kenney + Quaternius（CC0）；先实装模型。

---

## 一、模型实装（视觉多样化）

### 资源
全部 **CC0 1.0**（可商用、免署名）、低多边形、单材质（调色板贴图），适配 thin-instance 管线：

| 用途 | 资源 |
|---|---|
| 住宅 R | Kenney City Kit (Suburban) |
| 商业 C | Kenney City Kit (Commercial) |
| 工业 I / 仓储 | Quaternius 厂房仓库 + poly.pizza 工业件 |
| 道路（可选） | Kenney City Kit (Roads) |
| 车辆 | Quaternius LowPoly Cars / Kenney Car Kit（含货车） |

来源：kenney.nl、quaternius.com、poly.pizza（GLB 直载）。

### 技术管线
- **依赖**：`pnpm add babylonjs-loaders`（UMD 核心不含 glTF 加载器，需单独引入注册 `.glb/.gltf` 加载插件）。
- **加载策略**：`render/modelLoader.ts` 用 `SceneLoader.ImportMeshAsync` 异步加载每个 GLB；导入后 `MergeMeshes` 合并为单网格（Kenney/Quaternius 多为单材质，合并后仍可 thin-instance，保持每型 1 draw call）；归一化尺寸（footprint≈1 格）、朝向（+Z 朝路）、基座对齐 y=0；冻结材质。
- **构建链**：webpack 两份配置各加一条 `{ test: /\.(glb|gltf|png)$/, type: 'asset/resource' }`，`import url from './x.glb'` 得到 URL 后加载；模型文件放 `src/assets/models/`。
  （备选：运行时从 CDN 加载，省去打包，但依赖外部可用性与 CORS。）
- **异步启动**：`createScene` 改为 `async`，先 `await modelLoader.loadAll()` 再建场景与渲染；加载期间显示 DOM loading 界面。**任一模型加载失败自动回退到现有方块原型**，零阻塞、不崩。

### 渲染接入（多样化的关键）
- 扩展 `render/prototypes.ts` / `render/cityRenderer.ts` 的视觉 key：
  `bld:R:lvl2:var3` —— 每个「分区 × 等级（密度）」配 **2~4 个模型变体**。
- `cityRenderer.visualFor()` 选变体：按格子 idx 做确定性哈希 `(idx*2654435761>>>0)%variantCount` → 同一片区不再千篇一律，且可复现（与确定性模拟一致）。
- 每个变体 = 一个 `InstanceField` = 1 draw call。预计原型总数 ~60–90（建筑变体+设施+车辆+树），draw call 完全可控。

### 性能注意
- thin instances 对「多子网格 GLB」支持差 → 合并为单网格后再实例化。
- 多材质模型：要么按材质拆成多个共享矩阵的 field，要么优先选单材质资源（Kenney/Quaternius 满足）。
- 远景 LOD：相机拉远时高楼可退化为简化盒/纯色块（沿用阶段 0 LOD 思路），保证大城市帧率。

## 二、通勤逻辑层（统计行程 OD）

把阶段 1 的「重力流近似」升级为**显式分配 + 分时段行程流**——这是「住宅出来→去工厂/商店上班→去商店购物→回家」真实链路的逻辑内核。

### 持久分配（谁去哪）
- 每栋住宅的居民分配：**工作地**（最近可用的 C/I 岗位）+ **常去商店**（最近 C）。用路网 BFS 就近匹配，岗位/店容量满则顺延次近。
- 数据存于 `Building`：`workplaceId`、`shopId`；工作地维护 `workerHomes: number[]`。
- 建筑增删/路网变化时**增量重算**受影响分配（沿用脏标记哲学），不每日全量。

### 行程流与通勤时间
- 一天分三时段（由模拟时钟驱动）：早通勤 `home→work`、午购物 `home→shop`、晚归 `work→home`。
- 每段沿**缓存的最短路径**把通勤量压到路段 → 得到路段负载/拥堵（替换并增强现 `sim/traffic.ts`）。
- **通勤时间 = 路径上各路段拥堵的累加**。反馈：
  - 通勤过长 → 住宅幸福度↓、工作地生产力↓、商业到店客流↓。
  - 无路连通到工作地/商店 → 该住宅「找不到工作/买不到东西」→ 需求与满意度下降。
- 路网设计从此**真正影响城市运转**（堵车→商业衰退→税收降），契合硬核诉求。

### 性能
- 分配与路径**缓存**，仅拓扑变化时重算；行程流每游戏日推一次。
- 路径用路网 BFS（已具备 `sim/util.ts:bfsRoad`）；大城市可对 OD 抽样或按区聚合，封顶计算量。

## 三、车辆 agent 可视化（代表性车流）

在通勤逻辑之上，渲染「看得见的车流」，但**不 1:1 模拟每个市民**（浏览器扛不住）。

- `agents/vehicles.ts`：一个**代表性车辆池**（thin-instanced 车模型，封顶 ~300–500 辆）。
- 每辆车按通勤/货运的**真实路径 + 时段**实际行驶：逐帧沿路段序列插值移动，**速度 = 基速 /(1 + 路段拥堵)**，到达后取下一段行程。
- 车辆数量按**真实行程总量比例缩放**采样——看起来繁忙程度与实际交通相称，但规模恒定可控。
- 车辆是**纯表现层**：从通勤/货运系统取「行程订单」，不反向影响逻辑（逻辑由第二节的统计流负责）。货车复用同一系统（见第四节）。

## 四、三级产业供应链与货运

### 物料链（原料 → 工厂 → 商业 → 居民）
| 环节 | 角色 | 行为 |
|---|---|---|
| 原料 | 原料业（新分区/设施：农场/矿场/林场） | 用地/工产出**原料** |
| 工厂 | 工业 I | 消耗**原料** + 用工/电 → 产出**货物** |
| 商业 | 商业 C | 进**货物**入库 → 卖给居民 |
| 居民 | 住宅 R | 到店消费扣库存 |

- 每个生产/销售建筑带**库存缓冲**（原料库存、货物库存、成品出库）。
- 产出率 ∝ 在岗 × 生产力 × （电/水/原料是否到位）；缺上游投入则减产。
- 商店缺货 → 掉销售/掉满意；工厂无原料 → 停产；原料无工厂消化 → 减产。

### 货运（绑定交通）
- 物料沿路网配送：原料→工厂、工厂→商店，各生成**货运行程**，复用第三节车辆系统派**货车**。
- 无路连通或严重拥堵 → 配送受阻 → 下游缺料/缺货。**进口/出口**作为兜底与经济杠杆：短缺可花钱进口、过剩可出口换钱。

### 与现有系统耦合
- 接入 RCI 需求：货物供不应求压低 C 需求；工厂卖不出去压低 I 需求；原料短缺压低原料业以外的扩张。
- 接入经济：进出口、货运成本计入月结；产业链顺畅 → 税基与就业上升。

## 七、风险与取舍

| 风险 | 取舍 |
|---|---|
| 逐市民模拟过重 | 统计行程驱动逻辑，车辆只做代表性采样（封顶数百辆） |
| 多子网格 GLB 难实例化 | 合并单网格再 thin-instance；优先单材质资源 |
| 三级供应链平衡复杂 | 先用「库存缓冲 + 进出口兜底」保证不卡死，再调数值曲线 |
| 二进制资源打包 | webpack asset/resource 打包；失败回退方块，绝不阻塞启动 |
| 大城市帧率 | LOD + 分块剔除 + 车辆封顶 |
